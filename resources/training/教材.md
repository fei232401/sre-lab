# 第一部分：知识扫盲

## 第1章：什么是 AI Infra？

### 1.1 用大白话解释

假设你开了一家"智能客服公司"。你的客户（比如一家电商网站）会发来用户的问题，你需要用 AI 给出回答。

**问题来了**：
- AI 模型跑在哪里？（需要 GPU 服务器）
- 几百个客户同时提问怎么办？（需要排队、限流）
- 如果服务器挂了怎么办？（需要备用方案、自动切换）
- 怎么知道服务器状态？（需要监控、追踪、告警）
- 不同的问题该用哪个 AI 模型？（需要路由调度）

**AI Infra（AI 基础设施）就是解决以上所有问题的**。它不是训练 AI 模型，而是把训练好的模型**部署上线、对外提供服务、保证稳定运行**。

### 1.2 三个项目分别做什么？

| 项目 | 一句话 | 类比 | 核心能力 |
|------|--------|------|---------|
| AI Infra Gateway | 在普通电脑上搭建 AI 服务 | 自己家里开店 | 鉴权·限流·熔断·SSE 流式转发·GPU 仪表盘 |
| SRE-LAB | 用 K3S 搭建可扩展的 AI 平台 | 连锁店（随时加门店） | GitOps·HPA·Prometheus·Loki·ArgoCD |
| AI Model Scheduler | 统一管理多个 AI 服务，智能分配请求 | 总调度台 | 5种路由策略·异构后端·自适应评分 |

### 1.3 L1 实验：第一次调用 AI API

```bash
# 实验 1-1：用 curl 调 Ollama（本地推理引擎）
curl http://localhost:11434/api/tags
# 观察返回：{"models":[{"name":"qwen2.5:0.5b",...}]}

  -H "Content-Type: application/json" \
```

**思考题**：
1. `total_duration` 是纳秒，换算成秒是多少？
2. `eval_count / (total_duration/1e9)` 等于多少 token/秒？（这就是 Throughput）
3. 如果把 `stream` 改成 `true`，返回格式有什么不同？

## 第3章：核心概念速成

### 3.1 LLM 推理指标

| 指标 | 含义 | L4 实验你会测到 |
|------|------|------------|
| **TTFT** | Time To First Token：发请求到收到第一个字的时间 | Ollama ~3400ms, vLLM ~30ms |
| **TPOT** | Time Per Output Token：每个字的平均生成时间 | Ollama ~8ms, vLLM ~2ms |
| **Throughput** | 每秒生成 token 数（tokens/s） | Ollama ~198 t/s, vLLM ~5000+ t/s |

### 3.2 推理引擎对比

| 引擎 | 特点 | 本项目中 |
|------|------|---------|
| **Ollama** | 极简，一行命令启动 | Gateway 本地后端 |
| **vLLM** | 高性能，Continuous Batching | Scheduler 生产后端（RTX 4090） |
| **Mock** | 模拟低延迟 | Scheduler 教学演示 |

### 3.3 GPU 硬件基础

```
RTX 4060 Laptop (8GB VRAM)：开发环境，同时跑 0.5B + 1.5B 模型
RTX 4090 (24GB VRAM)：  生产环境，跑 7B 模型 + 64路并发

关键概念：
  VRAM = 显存，决定了能跑多大的模型
  CUDA Core = 计算单元，决定了推理速度
  Tensor Core = 矩阵加速单元，FP16 推理的主力
  Memory Bandwidth = 显存带宽，大模型推理的瓶颈通常是带宽而非算力
```

# 第二部分：项目一——AI Infra Gateway 精讲 + 动手实验

## 第4章：Gateway 架构总览

```
客户端 → :8000 Gateway
          ├── JWT HS256 鉴权（60min 过期 + API Key 回退）
          ├── Token Bucket 限流（容量10, 5 QPS 补充）
          ├── Circuit Breaker 熔断（3次失败→OPEN→30s→HALF_OPEN→恢复）
          ├── 2次指数退避重试
          └── SSE 流式逐 token 转发 → Ollama :11434 → GPU

Dashboard :9090  → pynvml 驱动级 GPU 采集（温度/显存/利用率/功耗）
OTEL Tracing   → 环境变量 OTEL_ENABLED=1 激活（TEMPO-ready）
```

### 4.1 L2 实验：理解鉴权中间件

```bash
# 实验 4-1：不带 token 访问——应返回 401
curl http://localhost:8000/health
# → {"status":"ok"}  （health 是公开路径）

curl http://localhost:8000/api/models
# → {"error":"Unauthorized"}  需要鉴权！

# 实验 4-2：带 token 访问
curl -H "Authorization: Bearer sk-infra-gateway-dev-key-2026" \
  http://localhost:8000/api/models
# → 返回模型列表
```

**理解要点**：`auth_middleware` 函数检查每个请求的 Header 中有没有合法 token。`/health` 和 `/docs` 在白名单中直接放行。

### 4.2 L2 实验：触发限流看 429

```bash
# 实验 4-3：快速连续发 20 个请求
for i in $(seq 1 20); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -H "Authorization: Bearer sk-infra-gateway-dev-key-2026" \
    http://localhost:8000/api/models
done
# 前 ~10 个返回 200，后面开始返回 429 Too Many Requests
# 这就是令牌桶在起作用：桶里只有 10 个令牌，用完就拒绝
```

**理解要点**：令牌桶容量=10，每秒补充 5 个。突发 10 个请求可以全部通过，持续超过 5QPS 就会触发限流。

### 4.3 L5 实验：看熔断器状态机

```bash
# 实验 4-4：观察熔断器状态

# → {"status":"ok","circuit_breaker":"closed"}

# 实验 4-5：模拟后端挂掉（停掉 Ollama）
# 然后连续发 3 个请求到 /api/models
# 观察日志输出：
#   [熔断器] 状态: OPEN (连续 3 次失败)
# 30 秒后自动尝试恢复：
```

### 6.2 L4 实验：对比四种路由策略

```bash
# 实验 6-1：延迟优先（vLLM 应该被选中——TTFT 最低）
curl -X POST http://localhost:9000/v1/chat/completions \
  -H "X-Routing-Strategy: latency" \
  -H "Authorization: Bearer test-key" \
  -d '{"model":"qwen2.5:0.5b","messages":[{"role":"user","content":"测试"}]}'

# 实验 6-2：成本优先（Ollama 应该被选中——免费）
curl -X POST http://localhost:9000/v1/chat/completions \
  -H "X-Routing-Strategy: cost" \
  -H "Authorization: Bearer test-key" \
  -d '{"model":"qwen2.5:0.5b","messages":[{"role":"user","content":"测试"}]}'

# 实验 6-3：会话亲和（同一 X-Session-Id 始终路由到同一后端）
for i in 1 2 3 4 5; do
  curl -s -X POST http://localhost:9000/v1/chat/completions \
    -H "X-Session-Id: my-session-123" \
    -H "Authorization: Bearer test-key" \
    -d '{"model":"qwen2.5:0.5b","messages":[{"role":"user","content":"hi"}]}' \
    | python3 -c "import sys,json;d=json.load(sys.stdin);print(d.get('_routing',{}).get('backend_id','?'))"
done
# 所有 5 次请求都应该返回同一个 backend_id
```

**理解要点**：Session Affinity 保证同一用户的多轮对话始终在同一后端上——后端可以复用 KV Cache，避免每次对话都从头计算。

### 6.3 L4 实验：改权重观察流量分配

```
TEMPO 是可观测性的黄金标准：

  T = Tracing  （分布式追踪）  → OpenTelemetry → 看"请求怎么流转的"
```

```bash
# 实验 7-3：运行 CUDA 精测
python ai-infra-gateway/04-infrastructure/cuda_profiler.py
# 输出：
# ============================================================
#   CUDA Profiler — NVIDIA GeForce RTX 4090 (8.9)
#   VRAM: 20.0GB used / 24.0GB total
# ============================================================
# [1/4] Memory bandwidth...
#   H2D: 28 GB/s  |  D2H: 27 GB/s  |  Device: 1008 GB/s
# [2/4] Peak FLOPS...
#   FP16: 82.580 TFLOPS  |  FP32: 41.289 TFLOPS
```

**各指标的含义**：
- **Memory Bandwidth ~1000 GB/s**：RTX 4090 的显存带宽。大模型推理的瓶颈通常是这个。
- **FP16 ~83 TFLOPS**：理论峰值算力。实际推理利用率通常只有 30-50%。
- **Attention 0.35ms**：一个 Transformer 层的注意力计算时间。7B 模型有 28 层，每层都要算。
- **vLLM TTFT 32ms**：比 Ollama 的 3400ms 快 ~100 倍——这就是 Continuous Batching + PagedAttention 的效果。

# 第六部分：CUDA 精测 + vLLM 生产调优

## 第8章：从 benchmark 到 profiler

### 8.1 benchmark vs profiler

| | benchmark（压测） | profiler（剖析） |
|---|---|---|
| 回答什么问题 | 系统能撑多少 QPS？ | 时间花在哪里了？ |
| 测量对象 | 端到端延迟、吞吐 | GPU kernel 执行时间、显存带宽 |

# 第七部分：面试实战

## 第9章：高频问题与标准回答

### Q1: "介绍一下你的 AI Infra 项目。"

**标准回答**：
"我搭建了一套从裸金属到云原生的 AI 推理基础设施体系。第一个 Gateway 在极端约束下徒手实现鉴权+限流+熔断+SSE 流式转发。第二个 SRE-LAB 用 K3S 搭建云原生 AI 平台，涵盖 Prometheus+Grafana+Loki+ArgoCD。第三个 Scheduler 实现 5 种路由策略，能统一调度本地 Ollama、K3S 集群和 vLLM 云端 GPU。三个项目均集成 OpenTelemetry 分布式追踪（TEMPO-ready），并配有 CUDA Profiler 做 GPU 精测。"

### Q2: "令牌桶和漏桶的区别？"

漏桶严格匀速，令牌桶允许攒令牌应对突发。LLM 推理是间歇性流量——平时没人用，突然来 10 个请求，令牌桶可以让 10 个一起通过，漏桶只能一个个放行。

### Q3: "SSE 和 WebSocket 的区别？"

SSE 是 HTTP 原生单向推送，WebSocket 是双向通道。LLM 推理只需要服务器→客户端推送 token，SSE 更合适。SSE 天然支持断线重连，实现更简单。

### Q4: "怎么保证高可用？"

三层防御：Gateway 层熔断器（3次失败→自动隔离）；K3S 层 HPA 自动扩缩 + PDB 优雅终止；Scheduler 层每 10s 健康检查自动摘除故障后端，多后端冗余。

### Q5: "可观测性怎么做？"

TEMPO 体系：Tracing（OpenTelemetry，看请求流转路径）、Metrics（Prometheus，看系统状态指标）、Profiling（CUDA Profiler，看 GPU 瓶颈）。三者配合实现全链路可观测。
